# Task 03 - Integration Design

I would architect the flow with the landing page submitting to a lightweight serverless endpoint, not directly from the browser to every downstream tool. The browser fires `window.dataLayer.push({ event: 'consultation_form_submitted', ... })` only after client-side validation passes, and then sends the form payload with `fetch()` to a backend endpoint such as a Node.js serverless function on Netlify, Vercel, or AWS Lambda. This backend becomes the single source of truth for processing the lead. From there, the endpoint first writes a log entry with timestamp, request ID, and payload hash for observability, then checks HubSpot before creating or updating a contact.

For HubSpot, I would use the **HubSpot CRM Contacts API**, not native form embed, Zapier, or Make. The reason is that this flow depends on phone-based identity, while HubSpot’s default deduplication is on email, not phone. Since the landing page intentionally keeps only Name and Phone, the backend should search HubSpot for an existing contact by normalized phone number first. If a match exists, update that contact with latest name, source = `Google Ads - Consultation Landing Page`, and lead status = `New Enquiry`; if not, create a new contact. Because HubSpot does not safely deduplicate on phone by default, the backend must own this logic explicitly.

After the HubSpot upsert succeeds, the backend should enqueue a WhatsApp confirmation request to Karix using their WhatsApp Business API and include a delivery deadline timestamp. In parallel, the landing page conversion should be sent to Google Ads through GA4 import, with GA4 receiving the `consultation_form_submitted` event from GTM. I would not trigger Google Ads only after WhatsApp delivery, because campaign optimization should depend on lead submission, not messaging success.

The single biggest failure point is **phone-based identity resolution in HubSpot**. If formatting is inconsistent, duplicate contacts can be created and downstream follow-up becomes messy. The fallback is strict phone normalization to E.164 format, pre-upsert lookup by phone, and an internal lead log keyed by normalized phone + date so failed or ambiguous records can be retried manually.

The WhatsApp 2-minute SLA can break because of API timeouts, queue backlog, invalid phone formatting, HubSpot latency, or Karix template rejection. I would monitor this with timestamped event logs for submission, HubSpot upsert, Karix request, and Karix response; create alerts if any lead has no WhatsApp API success within 90 seconds; and add a retry queue with exponential backoff plus an ops dashboard for pending and failed messages.
